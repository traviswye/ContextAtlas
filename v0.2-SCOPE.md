# ContextAtlas v0.2 Scope

> **Status:** Proposed. Informed by v0.1 completion state and Phase 5
> reference-run findings. Subject to refinement during Stream A execution
> if MCP disclaimer investigation or PyrightAdapter refinements surface
> unexpected scope.
>
> **Document purpose:** Make v0.2 concrete enough to start implementing
> while preserving evidence-gating for post-v0.1 decisions.

> - Potentially additional adapters (Go, Rust, C#)
> - Adapter conformance test suite
> - Benchmark extension to Python codebase

Reality at v0.2 planning time:

  against httpx on 2026-04-22).
- **Adapter conformance test suite shipped in v0.1** (commit 6f8d8ae,

- **httpx atlas produced and dogfooded** — Python benchmark extension
  substrate complete pending reference-run execution.
- **Phase 5 reference run complete** (benchmarks commit be65a96, hono).
  Efficiency thesis empirically supported on win-bucket prompts (50–71%
  tool-call reduction, 36–83% cost reduction); tie/trick buckets behave
  as RUBRIC predicted. Single-run methodology; quality axis deferred to
  step 13. Full synthesis at
  [`../ContextAtlas-benchmarks/research/phase-5-reference-run.md`](../ContextAtlas-benchmarks/research/phase-5-reference-run.md).

v0.1 delivered more than its original ROADMAP scope, and Phase 5
validated its core thesis in the efficiency direction. v0.2 therefore
re-scopes around what's actually ahead: generalization across
languages and across repos under the same measurement protocol.

## v0.2 thesis

**ContextAtlas works across languages and repos, not just the authors'
hand-picked TypeScript sample.**

Phase 5 validated the efficiency thesis on hono (win-bucket prompts,
TypeScript, single reference run). v0.2 tests whether that validation
holds across (a) a third language adapter and (b) the already-indexed
httpx Python repo under the same reference-run protocol.

The underlying question narrows from v0.1's "does the tool work?" to
v0.2's "does it generalize across the languages and repos it claims
to support?" External-user validation ("does it work on *someone
else's* codebase?") is a distinct thesis and moves to v0.3, where the
claim substrate is rich enough (Stream C) for that test to be
meaningful rather than testing a crippled version of the tool.

---

## Guiding principles (inherited)

All v0.1 guiding principles apply unchanged. The ones that most
constrain v0.2 decisions:

- **Evidence-gated upgrades.** No v0.2 feature ships without benchmark
  or dogfood evidence that it's needed. Applies especially to adapter
  additions.
- **Do it right, not descope for timeline.** v0.2 is not time-boxed.
  Scope is bounded by "what does 'works across languages and repos'
  require?" not by a calendar.
- **Dogfood always.** Each v0.2 feature dogfoods on ContextAtlas itself
  + at least one external benchmark target before it ships.
- **One-way migrations.** Any schema, ID format, or contract change
  lands as a deliberate version bump. No backward-compat cruft for
  v0.1 consumers.

## Scope

v0.2 delivers two capability streams plus cross-repo benchmark evidence.
Streams are sequenced (A → B) with httpx reference run landing between.

### Stream A — Adapter quality polish

**Motivation:** Python adapter shipped in v0.1 but surfaced known gaps
during httpx dogfood. Phase 5 additionally surfaced a Claude-Code-CLI
MCP interaction that damages user-facing output quality. Polish these
before expanding to new languages — fixing adapter bugs is cheaper
than building new adapters on top of bugs, and investigating
CLI-integration issues before external exposure (v0.3+) protects the
dogfood-trial story when it lands.

**Scope:**

1. **PyrightAdapter kind mapping refinements** (v0.1 backlog item #1)
   - `__all__` module list → `variable` (currently `class`)
   - Enum class members (CLOSED, UNSET, etc.) → `variable` or
     new `enum_member` kind (requires ADR-01 amendment if new kind)
   - Dunder methods (`__aenter__`, `__enter__`, etc.) → `method`
     (currently sometimes `class` via nested-children pass)
   - Post-filter algorithm documented in an ADR if new SymbolKind added

2. **Cost tracking in extraction pipeline** (v0.1 backlog item #2)
   - Accumulate `response.usage` from Anthropic SDK responses
   - Add `cost_usd`, `input_tokens`, `output_tokens` to
     `contextatlas index` stdout summary per ADR-12 contract
   - Same fields in `--json` output
   - Budget warning mechanism when run-level cost exceeds configured
     ceiling (new optional config field `extraction.budget_warn_usd`
     or equivalent `--max-cost` CLI flag; exact surface resolved
     during implementation)

3. **Unresolved candidate diagnostics**
   - Current pipeline reports counts (`unresolved_candidates=54`)
     but not which specific tokens failed to resolve
   - Add `--verbose` flag to `contextatlas index` that lists
     unresolved tokens with source claims
   - Helps users debug ADR-to-symbol drift

4. **TypeScriptAdapter parity check**
   - Conformance suite caught one TS bug in v0.1 (ENOENT on missing
     files). Worth a systematic pass:
   - Run the full Python-adapter dogfood query set against the
     TS adapter indirectly via hono atlas
   - Identify any Python-only affordances that should exist in
     TypeScript too (e.g., is the declaration-header parser's
     pathology handling as robust in TS?)

5. **Claude Code CLI MCP permission-disclaimer investigation**
   (from Phase 5 §4.3, §7.2)
   - Four of five beta-ca cells prefaced answers with *"I don't have
     permission to use the ContextAtlas tools"* despite clearly
     successful MCP calls in the trace. h1 beta-ca is the clearest
     example: model used 5 MCP responses to produce a correct
     ADR-01-citing answer while prefacing with a claim it lacked
     access.
   - Leading hypothesis: Claude Code interprets certain MCP response
     shapes as permission-denial signals. Secondary hypothesis:
     `--bare --strict-mcp-config` spawn-flag interaction.
   - **Timebox: 2 days (advisory, not enforced).** Two permitted
     outcomes:
     - (a) Root-cause and fix at our layer (spawn flags, response
       shape tweaks, or similar)
     - (b) Document as known Claude-Code-CLI interaction quirk in
       README with user-facing workaround or caveat. Becomes
       follow-up for v0.3+ if an upstream fix lands.
   - Either outcome closes the investigation; do not over-invest.

**Deliverables:**
- ADR-14 if new SymbolKind needed for enum members
- Refined PyrightAdapter with ~20–50 LOC post-filtering
- Cost tracking in anthropic-client + CLI summary
- Verbose diagnostics mode
- MCP disclaimer investigation outcome (fix PR *or* README
  documentation update)

**Estimated scope:** ~450–550 LOC + tests. ~2–3 focused days.

### Stream B — Additional language coverage + cross-repo benchmark

**Motivation:** v0.2's thesis requires evidence on more than one
language (Python already validated in v0.1 by extraction; not yet by
reference-run). The adapter conformance suite exists to prove the
plugin interface holds across languages; needs more than two data
points. And the Phase 5 methodology needs to demonstrate it
generalizes to a second repo before we claim cross-repo validity.

Stream B therefore has two independent but related sub-deliverables:
a third language adapter, and the httpx reference run (the missing
cross-repo datapoint for the existing Phase-5 methodology).

**Decision gates for additional language:** unchanged from previous
draft — portfolio demand + LSP maturity + benchmark target
availability.

**Proposed v0.2 additional language: Go via `gopls`.**

Rationale for Go over alternatives:
- Official Google-maintained LSP, stable behavior
- Major platform-engineering language (matches target job market)
- Reasonable-sized benchmark targets available (e.g., `caddy`,
  `cobra`, `viper`)
- Probe-before-code pattern from ADR-13 transfers directly
- No Protocol/Interface ambiguity — Go has explicit interfaces

Alternative: **Rust via `rust-analyzer`** if portfolio demand shifts.

**Scope:**

*Go adapter track:*
1. ADR-N documenting LSP choice + empirical probe findings
2. Probe artifacts (throwaway fixture + findings file)
3. `GoAdapter` implementation
4. Conformance suite passes for new adapter
5. Go benchmark target registration (ADRs, prompts, config, pinned
   clone) — one target: **cobra**.
   - Rationale: CLI framework internals are less common in LLM
     training data than HTTP routing, producing larger and more-
     attributable alpha-vs-CA deltas on win-bucket prompts — a
     stronger demonstration of the tool's value than a within-
     domain parallel would produce.
   - **Probe-phase fallback:** If the Go adapter probe reveals
     cobra has fewer than ~6 ADR-worthy architectural decisions
     (some CLI frameworks are architecturally "flat" — API surface
     design rather than composition/constraint decisions), swap to
     `gin` as the benchmark target. Probe artifacts are cheap and
     the fallback doesn't require rework elsewhere in Stream B.
6. Go dogfood extraction against pre-registered target

*httpx reference run track (Phase-5-style):*
7. Re-extract httpx atlas against Stream A's refined PyrightAdapter
   (kind-mapping fixes will change atlas output)
8. Execute reference run protocol against httpx using the locked
   httpx prompts and four conditions (alpha / ca / beta / beta-ca)
9. Commit artifacts to benchmarks repo matching Phase 5's
   `runs/reference/hono/` structure
10. Synthesis note comparing hono vs httpx efficiency patterns —
    does the win/tie/trick bucket behavior replicate cross-language?

**Estimated scope (Go adapter):** ~1500–2000 LOC + tests. ~3–4
focused days.

**Estimated scope (httpx reference run):** Execution-heavy, not
implementation-heavy. ~1 day to run + ~1 day to synthesize.

**v0.2 caps at one additional language.** Further languages (Rust,
C#, Java) are v0.3+ based on demand signal and v0.2's findings about
ADR-03 interface stability.

### Stream C — Claim source expansion [MOVED TO v0.3]

> **Status: Moved to v0.3 during v0.2 planning.**
>
> Rationale: Phase 5 evidence supports the thesis on win-bucket
> prompts where ADRs carry load; it does not mandate additional claim
> sources for v0.2's goals (cross-language + cross-repo validation).
>
> Stream D's external-user validation *does* require Stream C — but
> Stream D itself moves to v0.3, and Stream C moves with it so they
> land together. Decoupling them would mean either Stream D ships
> with a crippled claim substrate (testing the wrong thing) or
> Stream C ships without a consumer (building the wrong thing).
>
> Section retained below for traceability of the scoping decision;
> content migrates to v0.3-SCOPE.md when that document is created.

**Original Stream C scope** (preserved for traceability):

1. **JSDoc / docstring extraction for TypeScript and Python**
   - Parse JSDoc blocks above exports in TS
   - Parse docstrings in Python modules, classes, functions
   - Extract declarative statements as `soft` severity claims
   - Link to LSP symbol via existing ID format
   - Provenance: docstring at `file:line` (new claim source in schema)

2. **README / top-level docs mining**
   - If a `README.md` exists at repo root, extract architectural
     claims from sections matching heuristic headers (e.g.,
     "Architecture", "Design", "How it works")
   - Skip sections that look like usage or install docs

**Original scope estimate:** ~600–800 LOC + tests + schema migration.
~2–3 days.

### Stream D — External dogfood trial [MOVED TO v0.3]

> **Status: Moved to v0.3 during v0.2 planning.**
>
> Rationale: External users' repos don't generally have curated ADRs.
> With Stream C (docstring / README claim mining) deferred to v0.3,
> external atlas generation would produce near-empty atlases on
> typical repos. That tests a partial, unrepresentative version of
> the tool rather than its real value proposition. Running Stream D
> in v0.3 alongside Stream C ensures the first external trial hits
> the tool as it's actually intended to work.
>
> Section retained below for traceability of the scoping decision;
> content migrates to v0.3-SCOPE.md when that document is created.

**Original Stream D scope** (preserved for traceability):

1. Identify one external trial candidate (friend/coworker,
   permissively-licensed repo, or r/programming/HN solicitation).
2. Produce a "try this" artifact (install/setup instructions,
   example config, known limitations upfront).
3. Capture feedback structured (setup failures, wrong/missing
   claims, query outcomes, feature gaps).
4. Commit learnings as issues or `docs/external-trial-N.md` writeup.

**Original scope:** Mostly social / documentation. Low code cost.

## Beyond v0.2 scope

Explicit deferrals with reasoning (beyond items covered by ROADMAP.md's
v0.3+ sections):

**h5-class tie-bucket gap (TS-compiler-space prompts).** Phase 5 §3.2
documented that h5 loses because type-parameter flow lives in the TS
compiler's type graph, not in the architectural-intent layer.
Docstrings don't fix this; README mining doesn't either; nor does
cross-language adapter work. Addressing it requires either enriched
LSP structural signals (capturing type-graph slices) or a new signal
layer — neither of which is scoped into v0.3 (claim source expansion
+ external trial). This is a **v0.4+ investigation**, explicitly not
a v0.2 or v0.3 deliverable. Flagging here so future-me doesn't
retroactively expect v0.2 or v0.3 to improve h5-class benchmark cells
— neither version targets this gap. The natural home is alongside
the v0.4 semantic-layer work, where richer structural signals are
already on the table.

**Step 13 (blind grading, 3-run medians, full 24-prompt benchmark
expansion).** Per Phase 5 §7.5, step 13 runs *after* v0.2 ships. Its
purpose is quality-axis measurement — Phase 5 only measured efficiency.
Projected cost ~$176–210 based on observed Phase 5 spend. Step 13 is
its own undertaking, not folded into v0.2 execution.
- **Gating decision: single execution post-v0.3.** Rationale:
  signal-fusion maturity (v0.3's Stream C work) makes quality-axis
  measurement informative in ways the ADR-only substrate doesn't.
  Running step 13 post-v0.2 on current substrate produces a baseline
  we'd likely re-run after v0.3 anyway — paying ~$200 twice to learn
  the same thing is wasteful.
- **Option preserved: decomposition run at v0.2-state if v0.3 data
  surprises us.** If v0.3's step-13 output shows unexpected regression
  on specific prompts, or ambiguity about whether improvements came
  from v0.2 polish vs. v0.3 signal fusion, we can decompose by
  executing a third reference run at v0.2-state (git checkout to v0.2
  ship commit, run reference protocol against the v0.2 atlas, compare).
  This isolates the v0.2-effect from the v0.3-effect cleanly when
  needed, without pre-committing to ~$400 of measurement now.

**All other v0.2+ items preserved from ROADMAP.md** remain deferred
(embedding layer v0.4, task-shaped queries v0.5, hot-path caching
v0.6+, claim capture from agent sessions v0.6+, etc.).

## Sequencing

Streams have natural dependencies constraining parallelism:

- **Stream A → Stream B.** Do not build Go adapter on top of
  unresolved PyrightAdapter kind-mapping bugs. Stream A items 1–4
  complete before Stream B begins in earnest.
- **Stream A item 5 (MCP disclaimer investigation) is parallelizable**
  — can run concurrently with Stream B probe or adapter work. It's
  timeboxed and has no dependency on the other Stream A items.
- **httpx reference run gates on Stream A completion.** Stream A's
  PyrightAdapter refinements change atlas output; the httpx reference
  run should measure against the *refined* adapter, not the v0.1
  adapter. Sequence: finish Stream A → re-extract httpx atlas → run
  reference protocol.
- **httpx reference run technically independent of Go adapter work.**
  Run it between Stream A completion and Go adapter completion — a
  natural spot to break up the work and provides the cross-repo
  evidence before committing the rest of Stream B.
- **Step 13 gates on v0.2 ship.** Do not start step 13 execution
  before v0.2 completes. Step 13's cost envelope is too large to
  absorb into v0.2 and would delay the v0.2 ship gate.

**Reasonable execution order:**

1. Stream A items 1–4 (PyrightAdapter + cost tracking + diagnostics +
   TS parity)
2. Re-extract httpx atlas against refined PyrightAdapter
3. httpx reference run + synthesis note (Stream B sub-deliverable)
4. Stream A item 5 (MCP disclaimer investigation) — can run in
   parallel with step 5 below
5. Go adapter probe + ADR-N + implementation + conformance
6. Go benchmark target registration + dogfood extraction + Go
   reference run (MVP scale)
7. v0.2 ship
8. (Post-v0.2) Step 13 executes

## Cost envelope

v0.2 splits API spend across two budgets:

**Main-repo extraction budget (~$10–15):**
- Re-extraction runs during Stream A development (5–8 full
  self-extractions): ~$5–8
- httpx re-extraction against refined PyrightAdapter: ~$2–3
- Go adapter dogfood extraction on benchmark target (one-time):
  ~$2–3
- Contingency: ~$2

**Benchmarks-repo reference runs (~$15–25):**
- httpx reference run (24 cells at Phase-5 blended rates of
  $0.61/cell): ~$15
- Go benchmark reference run at MVP scale (6 prompts × 4 conditions,
  similar to Phase 5): ~$10–15. If alpha-condition prompts dominate
  cost and the Go benchmark uses fewer alpha-heavy prompts, this
  trends lower.

**Combined v0.2 total:** ~$25–40. Still well under step 13's
projected $176–210.

If the cost-tracking budget warning from Stream A #2 fires during
v0.2 work, treat that as a signal to investigate rather than raise
the cap.

## Success criteria

v0.2 ships when:

1. **Stream A complete:** PyrightAdapter refinements shipped, cost
   tracking in CLI output, verbose diagnostics mode, TS parity check
   complete, MCP disclaimer investigation landed (fix OR
   documented-as-known-issue).
2. **Stream B Go track complete:** Go adapter shipped with conformance
   suite passing, Go benchmark target registered (ADRs + prompts +
   config + pinned clone), Go dogfood extraction validated, Go
   reference run artifacts committed.
3. **Stream B httpx reference run complete:** Artifacts committed in
   benchmarks repo with Phase-5-comparable data shape; synthesis note
   committed comparing hono vs httpx efficiency patterns.
4. **Benchmark methodology demonstrated across three languages:**
   hono + httpx + Go benchmarks produce comparable data shapes;
   data-collection pipeline works identically across adapters; no
   per-language special-casing in the harness itself.

All four criteria are testable via committed artifacts. No criterion
is attempt-based (Stream D's attempt-based criterion moved to v0.3).

## Rescope conditions

v0.2 rescopes (not kills — rescopes) if:

- **Go adapter probe reveals fundamental LSP incompatibility with the
  ADR-03 interface.** Unlikely given gopls quality, but if found, pivot
  Stream B to Rust or defer the Go track entirely. The probe-before-code
  pattern from ADR-13 exists specifically to surface this early, before
  adapter implementation sunk-cost locks in.

- **Stream B benchmark fails to replicate Phase 5's efficiency
  pattern.** If the Go reference run shows no win-bucket improvement,
  v0.2 pivots from "ship Stream B" to "investigate why the pattern
  didn't replicate before shipping." The v0.2 thesis is cross-language
  generalization, and a non-replicating benchmark is the rescope
  trigger, not a demo failure to paper over.

- **httpx reference run contradicts hono Phase 5 findings.** If
  cross-repo validation fails on the Python side, v0.2 stops and
  investigates before adding Go. Same principle as above, on the
  earlier-in-sequence deliverable.

- **Stream A investigations surface deeper adapter issues than
  anticipated.** If PyrightAdapter refinements expose structural
  problems requiring ADR-03 amendments, v0.2 absorbs the architectural
  work rather than shipping the workaround.

Per ROADMAP.md's `Rescope conditions` section — same philosophy,
applied to v0.2 scope.

## Open questions (to resolve during v0.2)

1. **Is the ADR-03 plugin interface stable after three adapters, or
   does Go surface new requirements?** Answer shapes v0.3 priorities.
   Specifically: does the conformance suite need expansion for Go, or
   does Go fit within the current behavioral contract?

2. **Is "benchmark target pre-registration" scaling?** Three languages
   × custom targets × custom prompts is still linear per-addition. At
   what point does benchmark tooling need to support "run against
   arbitrary repo"? Probably v0.3+ decision, but worth flagging if
   Go benchmark registration feels especially burdensome.

3. **How does v0.2 handle multiple languages in one repo?** (e.g., a
   TypeScript + Python project). v0.1's config is single-language.
   Stream B's Go adapter doesn't force this decision, but ContextAtlas
   itself becomes a multi-language candidate once the dogfood atlas
   includes adapter subprocess-spawning code. Defers cleanly to v0.3+.

4. **Adapter plugin externalization.** With three adapters in-tree by
   end of v0.2, pressure increases. Probably v0.3 decision, but worth
   revisiting after Go adapter ships: is the friction of adding #4
   (say, Rust) higher than the friction of externalizing the interface?

5. **v0.3 timing trigger.** v0.2 defers Streams C and D. What's the
   signal that v0.3 should start? Candidates:
   - (a) **Evidence-based:** Stream B dogfood surfaces an
     ADR-dependency limitation concretely (e.g., Go benchmark target
     has sparse ADR coverage and produces a weak atlas), confirming
     Stream C is needed.
   - (b) **Calendar-based:** A fixed window post-v0.2 ship.
   - **Default preference: (a)**, consistent with the evidence-gated
     guiding principle. Calendar-based triggers are fallbacks if
     evidence doesn't arrive organically within a reasonable window.
