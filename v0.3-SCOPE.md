## v0.3 thesis

**Atlas extraction sharpens — claim attribution + ranking precision,
first non-ADR claim source, methodology hardening.**

Phase 5/6/7 validated the cross-language thesis on three reference
runs. v0.3 tests whether the atlas-as-team-artifact substrate is
*precise enough* to support more claim sources (docstrings) and
multi-symbol queries (Phase 7 §5.1) without amplifying the muddy-
bundle problem Phase 6 §5.1 identified. The underlying question
narrows from v0.2's "does it generalize?" to v0.3's "does it
sharpen?"

Quality-axis measurement (blind grading) remains v0.4 scope. v0.3
ships precision improvements + first non-ADR source + methodology
hardening; v0.4 measures whether those improvements move the
quality needle. Both versions are mandatory before v1.0.

## Guiding principles (inherited)

All v0.1/v0.2 guiding principles apply unchanged. The ones that
most constrain v0.3 decisions:

- **Evidence-gated upgrades.** Every v0.3 deliverable cites a
  v0.1/v0.2 finding or backlog note that triggered it. No
  speculative scope.
- **Do it right, not descope for timeline.** v0.3 is medium-to-
  large (1–2 weeks elapsed). Scope is bounded by "what does
  'extraction sharpens' require?" not by a calendar.
- **Dogfood always.** Each v0.3 deliverable dogfoods on
  ContextAtlas itself + measures impact via the v0.3 reference
  run before shipping.
- **One-way migrations.** Atlas schema v1.2 → v1.3 (or higher)
  follows ADR-11's additive pattern. Symbol ID format remains
  ADR-01 stable. No backward-compat cruft for v0.2 consumers.
- **Protect the primitive.** Theme 1.1 (multi-symbol
  `get_symbol_context`) and Theme 1.2 (claim attribution +
  ranking) both touch the primitive. Regressions there cascade;
  test coverage gates ship.

## Scope

v0.3 delivers four streams. Streams A and C are sequenced
prerequisites for Streams B and D respectively (see Sequencing
section below).

### Stream A — Atlas precision [HEADLINE]

**Motivation:** Phase 6 §5.1's muddy-bundle problem is the
load-bearing constraint on Stream B's value. Adding more claim
sources without sharpening per-symbol attribution + ranking
would degrade `get_symbol_context` bundle quality on
high-claim-count symbols. Phase 7 §5.1's grep-ceiling
finding on Go is closeable by a multi-symbol call shape that
also benefits TS/Python on dispersed-symbol questions. v0.3's
headline lives here — get the substrate right, then add to it.

**Scope:**

1. **Theme 1.2 — claim attribution + ranking precision** (filed
   at [`../ContextAtlas-benchmarks/research/atlas-claim-attribution-ranking.md`](../ContextAtlas-benchmarks/research/atlas-claim-attribution-ranking.md))
   - **Fix 1 — ADR authoring validation** (~30–50 LOC). At
     extraction time, flag frontmatter-declared symbols that
     don't resolve. Phase 6 ADR-05 example: 10 of 14
     frontmatter symbols don't resolve. Warning at extraction
     pushes authors toward resolvable symbol paths. Smallest
     intervention; ships regardless of Fix 2/3 outcomes.
   - **Fix 2 — narrower claim attribution** (~20–30 LOC).
     Drop frontmatter-baseline inheritance in
     `writeClaimsForFile`. Attach claims only to their
     claim-specific resolved candidates. Loses some safety-net
     coverage but sharpens per-symbol ranking. Evidence-gated:
     Stream D reference run validates whether the narrower
     attribution actually improves bundle quality before this
     ships outside an experimental flag.
   - **Fix 3 — symbol-scoped claim ranking** (~150–250 LOC,
     architectural). Rank claims within a `get_symbol_context`
     bundle by relevance. Two sub-approaches:
     - BM25 on claim text vs caller's query string (requires
       MCP tool-interface change to receive query). Smaller.
     - Embedding-based ranking (overlaps v0.4 semantic layer
       scope). Larger; defer to v0.4 unless v0.3 evidence
       forces.
   - **Choice between Fix 2 and Fix 3 evidence-gated.** Stream
     D's pre/post comparison on Phase 6 p4-stream-lifecycle
     decides. Fix 1 ships regardless; Fix 2 OR Fix 3 ships
     based on evidence; both is acceptable if scope permits.
   - **ADR-09 amendment likely needed** if Fix 3 BM25 path
     selected (extends ADR-09's scope from `find_by_intent`
     to `get_symbol_context`).

2. **Theme 1.1 — multi-symbol `get_symbol_context` call shape**
   (Phase 7 §5.1; ROADMAP-promoted)
   - Add multi-symbol variant: `get_symbol_context([sym1,
     sym2, ...])` returning a fused bundle covering N
     symbols. Closes the Go grep-ceiling gap by allowing a
     single MCP call to retrieve related symbols that
     knowledgeable Grep would have caught with regex
     disjunction.
   - Output format: per-symbol sub-bundles within a single
     compact text response. JSON variant follows ADR-04's
     opt-in pattern.
   - **MCP tool-interface change** — new ADR-N (or ADR-09
     amendment if BM25 path also lands) documenting the
     multi-symbol shape. CLAUDE.md "What to ask the user
     about" lists "Changing the MCP tool interface shape" —
     this lands deliberately.
   - Cross-language benefit: TS/Python prompts that span
     multiple symbols (e.g., "what's the contract between
     `Validator` and `Context.req.valid`?") can use the
     batched call shape too.

3. **Theme 1.3 — contextatlas commit SHA in atlas.json** (filed
   at [`../ContextAtlas-benchmarks/research/atlas-contextatlas-commit-sha-gap.md`](../ContextAtlas-benchmarks/research/atlas-contextatlas-commit-sha-gap.md))
   - Atlas schema bumps 1.2 → 1.3 (additive — same pattern as
     1.1 → 1.2 in v0.2 Step 9). New optional field
     `generator.contextatlas_commit_sha` captures
     `git rev-parse HEAD` at extraction time.
   - ADR-06 small amendment for the new field.
   - ~50–100 LOC including tests; small tagalong with the
     larger Stream A schema work.

**Deliverables:**
- Refined extraction pipeline with Theme 1.2 Fix 1 (and Fix 2 or
  3 per Stream D evidence)
- New multi-symbol `get_symbol_context` MCP tool variant + ADR
- Atlas schema v1.3 with `contextatlas_commit_sha`
- ADR-09 amendment if BM25-on-symbol-context path selected
- Test coverage on the primitive's new code paths

**Estimated scope:** ~600–900 LOC + tests, depending on
Theme 1.2 fix selection. ~4–6 focused days.

### Stream B — First non-ADR claim source: docstrings

**Motivation:** v0.1 + v0.2 indexed claims from ADRs only. Most
real repos have architectural signal in docstrings — JSDoc on
TypeScript exports, Python docstrings, Go doc comments. Adding
this as the first non-ADR source proves the multi-source
pipeline shape; remaining sources (README / PR / commit
messages) are mechanical extensions of the same pattern,
deliberately deferred to v0.4 to bound v0.3 scope.

The choice of "one source, not all four" is intentional:
v0.3's headline is *precision*, not *breadth*. Stream A's
attribution + ranking work is the load-bearing investment;
Stream B validates that the pipeline accepts a second source
shape without architectural surprise. v0.4 then scales breadth
once the precision substrate proves itself.

**Scope:**

0. **Cross-language probe phase (precedes implementation).**
   Before scaffolding extraction code in any specific language,
   examine the docstring surface across all three target
   languages — particularly Go's gopls hover output (richest
   structured markdown per ADR-14's §"Hover output is rich"
   finding). The contract for "what claim shape do we extract
   from a docstring?" must accommodate the most-structured
   surface (Go) so that less-structured surfaces (TS / Python)
   retrofit cleanly rather than the reverse. v0.2 retrospective
   §"Pre-implementation surveys caught hidden work before it
   bit" plus the ADR-13/14 probe-then-decision precedent both
   apply: probe first, lock contract, then implement.

   **Stream B sub-decision: language implementation order.**
   Two viable orderings, decided at execution time after the
   probe:
   - **(A) Go first** — implement against the richest surface
     first; TS / Python retrofit. Best fit if the probe shows
     Go's hover output materially shapes contract design.
   - **(B) TS first with prior probe** — implement against the
     simplest surface first after the probe has shaped the
     contract. Best fit if the probe shows the contract is
     surface-agnostic and TS's lower richness doesn't constrain
     it.
   Default: **(A) unless probe rules it out.** Mirrors the
   Step 8 ADR-14 lesson where Go-first probe-then-decide
   surfaced two non-obvious gopls runtime prerequisites that
   would have been retrofits had TS-first been the order.

1. **Docstring extraction — TypeScript (JSDoc)**
   - Parse JSDoc-style comments adjacent to exported symbols
     via the existing TypeScriptAdapter's symbol surface
   - Extract claim text from `@description`, `@remarks`,
     `@deprecated`, free-form leading prose
   - Severity inference: `@deprecated` → `hard`; explicit
     "must" / "MUST" / "required" in prose → `hard`; default
     `soft`
   - Symbol-keyed: claim attaches to the documented symbol's
     `SymbolId`. Mechanical because docstrings sit adjacent
     to their target

2. **Docstring extraction — Python**
   - Triple-quoted docstrings (PEP 257) at function / class /
     module level via PyrightAdapter substrate
   - Extract from `"""..."""` blocks; respect Google /
     NumPy / Sphinx style conventions (parse `Notes:`,
     `Warnings:`, `Deprecated:` sections explicitly)
   - Severity inference parallel to TS
   - Module-level docstrings keyed to a synthetic
     module-level SymbolId (TBD shape — likely
     `sym:py:<path>:<module>` or a new `(module)` reserved
     name)

3. **Docstring extraction — Go**
   - Doc comments preceding exported declarations (Go
     convention: comment block immediately above the
     `func` / `type` / `var` keyword)
   - Extract via gopls hover (rich markdown surface, ADR-14
     §"Hover output is rich" finding) or via direct comment
     parsing
   - Severity inference parallel to TS/Python
   - Build-tagged-file docstrings: surface from all files
     regardless of active tag (ADR-14 Decision 8 pattern;
     pipeline dedup is downstream concern)

**Note on items 1/2/3 ordering:** items are listed in
identifier order (TS / Python / Go), not implementation
order. Implementation order resolves at execution time per
Stream B sub-decision item 0 above.

4. **Source field convention**
   - Atlas claim entries gain `"source": "docstring:<path>"`
     pattern alongside existing `"source": "ADR-NN"` —
     extends the existing string field; no schema change
   - Provenance preserved: `source_path` is the file
     containing the docstring; `source_sha` from the
     containing file's SHA
   - Docstring claims and ADR claims compose at query time —
     `get_symbol_context` returns both kinds in the bundle,
     ranked per Theme 1.2 (this is why Theme 1.2 must
     precede Stream B)

5. **Extraction prompt update**
   - `EXTRACTION_PROMPT` in `src/extraction/prompt.ts` is
     ADR-tuned. Docstring extraction may need a sibling
     prompt tuned for shorter, less-structured input
   - Prompt frozen per ADR-02; if a new prompt lands,
     ADR-02 amendment documents the dual-prompt pattern
   - Pre-validate on 10–15 docstring examples across the
     three languages before scaffolding the pipeline change

**Deliverables:**
- Docstring extraction working end-to-end for TS, Python, Go
- Atlas claim entries with `source: "docstring:<path>"`
  surfacing alongside ADR claims
- ADR-02 amendment if dual-prompt pattern adopted
- Updated extraction-pipeline tests covering the new source

**Estimated scope:** ~500–700 LOC + tests + prompt
calibration. ~3–4 focused days.

### Stream C — Methodology hardening

**Motivation:** Theme 2.1 (atlas-file-visibility filter) is
mandatory before any v0.3 reference matrix. Theme 2.2
(cross-harness asymmetry tracking) and Theme 2.3 (Go-specific
cost priors) are additive improvements that compose with
Stream D's reference run. Three-pack with weak coupling —
Theme 2.1 first; 2.2 and 2.3 anywhere before Stream D.

**Scope:**

1. **Theme 2.1 — atlas-file-visibility methodology fix**
   (Phase 7 §5.2; ROADMAP-promoted)
   - **Source note still to author.** Per the v0.2
     retrospective lesson, author this during Stream C
     execution at
     `../ContextAtlas-benchmarks/research/atlas-file-visibility-benchmark-methodology.md`
     while the context is loaded — not after Stream D
     surfaces another instance.
   - **Recommended fix path: trace-time filter.** Detect
     cells where beta's trace references atlas paths
     (`atlases/<repo>/atlas.json`, `atlases/<repo>/index.db`,
     etc.) and exclude those cells from beta-vs-beta-ca
     delta tables. Lowest-cost iteration.
   - Alternative paths (clean-workspace mode, atlas-aware
     prompts) documented in the methodology note for v0.4+
     consideration if trace-time filter proves insufficient.
   - **Implementation in benchmarks repo**, not main repo:
     `src/harness/run.ts` filter pass + RUBRIC.md amendment.

2. **Theme 2.2 — cross-harness asymmetry tracking** (Phase 7
   §5.3; ROADMAP-promoted)
   - Synthesis-doc shape change: every v0.3+ reference run
     synthesis includes the explicit beta-ca-vs-beta vs
     ca-vs-alpha comparison table.
   - No code change. Adds rigor to whether the Phase 7 §5.3
     hypothesis ("CA delivers larger gains in CLI harnesses
     than in SDK harnesses") confirms or falsifies on
     additional reference-run targets.
   - RUBRIC.md amendment documenting the comparison as
     standard methodology.

3. **Theme 2.3 — Go-specific cost-prior bucket scale**
   (extracted from Phase 7 §8.4 during the backlog walk)
   - Refactor `COST_PRIORS_V0_1` (or its v0.3 successor) in
     `src/harness/run.ts` to support per-language bucket
     scales.
   - Seed Go priors from Phase 7 cobra data (~$0.30/cell
     blended vs hono+httpx ~$0.61/cell). Phase 7 cost data
     is sufficient; revised step-13 budget envelope of
     $115–150 (vs prior $176–210) feeds future scope
     planning.
   - RUBRIC.md amendment documenting per-language calibration
     pattern.

**Deliverables:**
- Trace-time filter shipped + atlas-file-visibility
  methodology note authored
- Cross-harness asymmetry comparison table standard in
  Stream D synthesis
- Go-specific cost priors seeded; per-language calibration
  pattern documented
- RUBRIC.md amendments for the methodology additions

**Estimated scope:** ~150–250 LOC (mostly in benchmarks repo;
small main-repo touchpoints if priors live there). ~1–2
focused days.

### Stream D — v0.3 reference run

**Motivation:** Stream A (precision) and Stream B (docstrings)
both claim quality improvements. Stream D measures them
against an apples-to-apples baseline.

**Apples-to-apples framing — load-bearing.** Stream D uses the
**SAME pre-registered prompt sets** (`prompts/hono.yml`,
`prompts/httpx.yml`, `prompts/cobra.yml`) that drove Phase 5,
Phase 6, and Phase 7. **Not new prompts.** This produces the
cleanest signal v0.3 can deliver: per-cell deltas between v0.2
atlases and v0.3-sharpened atlases on identical questions.
Where deltas favor v0.3, that's evidence Stream A and Stream B
work landed; where deltas don't favor v0.3, that's diagnostic
data about which improvements didn't materialize.

Comparable methodology to Phase 5/6/7 (single-run, three-
language); rigorous quality measurement (blind grading)
remains v0.4 scope.

**Scope:**

1. **Re-extract atlases for hono / httpx / cobra** against
   sharpened pipeline (Stream A + Stream B shipped). Same
   atlas-as-team-artifact pattern; new schema (v1.3 with
   `contextatlas_commit_sha`); docstring claims included
   alongside ADR claims.

2. **Re-run reference matrix on the same locked prompt sets.**
   `prompts/{hono,httpx,cobra}.yml` are the v0.2 pre-registered
   sets, untouched. Same 24 cells per repo, same four
   conditions, same single-run methodology. Run against the
   v0.3 atlases. Per-cell deltas vs Phase 5/6/7 baselines are
   the headline measurement.

3. **Pre-registration discipline preserved — and now
   load-bearing.** Existing prompt sets are locked per the v0.2
   pre-registration policy; v0.3 reference run **cannot** modify
   prompt text. New comparison tables (Theme 2.2) and the
   trace-time filter (Theme 2.1) are methodology additions, not
   prompt changes. If a prompt change becomes necessary during
   v0.3 execution, that's a scope decision worth surfacing
   explicitly — not a quietly-applied edit; the apples-to-apples
   framing depends on the locks holding.

4. **Synthesis note: Phase 8 v0.3 reference run.**
   - Mirrors Phase 5/6/7 structure. Phase 8 **must document
     at minimum** these three findings — each measures one of
     v0.3's headline claims:
     - **Theme 1.2 fix validation** — pre/post comparison on
       Phase 6 p4-stream-lifecycle (the cell that surfaced
       the muddy-bundle problem). Did the ranking fix
       improve bundle precision? Quantitative: top-claim
       relevance to query; qualitative: answer text changes.
     - **Stream B docstring source value** — does adding
       docstring claims improve win-bucket cells without
       amplifying tie/trick over-engineering? Compare
       per-cell metrics across v0.2 atlas (ADR-only) vs v0.3
       atlas (ADR + docstrings).
     - **Theme 1.1 multi-symbol API exercise** — at least
       one prompt category benefits from the batched call
       shape (target: re-evaluate cobra c4-stream-lifecycle
       where Phase 7 §5.1 surfaced the grep-ceiling). Does
       the Go grep-ceiling close materially?
   - **Additional findings allowed.** Same way Phase 7's
     three findings (§5.1 / §5.2 / §5.3) emerged from
     measurement rather than from pre-specification, Phase
     8 may surface additional findings during Stream D
     investigation. The minimum-three list above is a floor,
     not a ceiling — synthesis-note LOC is finding-scaled
     per the v0.2 retrospective lesson.

5. **Quality-axis caveat preserved.** Phase 8 measures
   efficiency + bundle precision (via Theme 1.2 validation).
   Correctness scoring, hallucination rates, and constraint-
   violation detection remain v0.4 scope (blind-graded full
   benchmark expansion).

**Deliverables:**
- Three v0.3 atlases committed (hono / httpx / cobra)
- Reference run artifacts in `runs/reference/<target>/` for
  each
- Phase 8 synthesis at
  `../ContextAtlas-benchmarks/research/phase-8-v0.3-reference-run.md`
- Cross-harness asymmetry comparison table populated for the
  first time per Theme 2.2

**Estimated scope:** ~$15–25 in API spend (matches v0.2
reference-run envelope; three matrices at ~$7–10 each per
revised cost priors). ~1–2 focused days for execution +
synthesis.

## Beyond v0.3 scope

The following are deliberately deferred. Each lands in a
specific later version with rationale.

**v0.4:**
- **Quality-axis measurement (blind grading).** Step-13 full
  benchmark expansion, post-v0.3. v0.3's Stream D is
  efficiency + bundle-precision measurement only.
- **Remaining claim sources.** README / `docs/` /
  CONTRIBUTING.md parsing, git commit-message extraction
  (beyond v0.1's regex), PR description mining (opt-in).
  Mechanical extensions of Stream B's pipeline once the
  docstring source proves itself.
- **Semantic layer (vector embeddings).** Per ADR-09's
  evidence-gated upgrade ladder, only if Theme 1.2 Fix 3
  BM25 path proves insufficient on Phase 8 evidence. v0.3's
  Theme 1.2 Fix 3 sub-decision feeds this directly.
- **Theme 3.1 — budget-prompt enhancement.** Operator
  interactivity for cost-ceiling decisions; trigger is
  step-13 scale, not v0.3.

**v0.5+:**
- **Task-shaped bundle queries** (`why_does_this_fail`,
  `onboard_to_feature`, `audit_change`) — per ROADMAP, lands
  where the headline efficiency-collapse story lives.
- **Adapter additions beyond Go.** Rust, .NET, Java per
  demand. Orthogonal to v0.3 thesis.
- **Self-generation.** v1.0 territory: ContextAtlas helps
  generate its own next-version improvements via the atlas
  it produces of itself. v0.3 + v0.4 are mandatory
  prerequisites; not v0.3 work.

**Out of all roadmap scope:**
- Web dashboard / VS Code extension — separate product
  surfaces, not the MCP server.
- General LSP wrapper — not the product.

## Sequencing

Streams have natural dependencies constraining parallelism:

- **Stream A 1.2 (claim attribution + ranking) must precede
  Stream B (docstrings).** Adding docstring claims without
  the ranking fix amplifies the muddy-bundle problem Phase 6
  §5.1 identified (docstrings naturally cluster around their
  target symbols, multiplying claims-per-symbol). This
  sequencing is *load-bearing*, not a preference. Surfaced
  explicitly in
  [`../ContextAtlas-benchmarks/research/v0.3-backlog-inventory.md`](../ContextAtlas-benchmarks/research/v0.3-backlog-inventory.md)
  §"Sequencing constraint."

- **Stream A 1.1 (multi-symbol API) is independent of
  Stream B.** Can ship in parallel with Stream B work or
  before Stream A 1.2 lands. Practical order: bundle with
  the ADR-N work for the new tool variant alongside other
  Stream A items.

- **Stream A 1.3 (commit SHA in atlas) tagalongs.** Schema
  change pairs cleanly with whatever schema work Stream A
  1.2 needs (if any) or ships independently — small enough
  to slot anywhere.

- **Stream C 2.1 (atlas-file-visibility filter) must precede
  Stream D.** Without it, c6-class measurement artifacts
  contaminate beta-condition data on the v0.3 reference run.
  This is *also* load-bearing.

- **Stream C 2.2 (cross-harness tracking) and 2.3 (cost
  priors) are independent.** 2.2 lands at synthesis time;
  2.3 lands before cost projections need to be defensible.
  Either anywhere before Stream D ships.

- **Stream B → Stream D.** Docstring extraction must produce
  v0.3 atlases before the reference run measures them.

**Reasonable execution order:**

1. Stream A 1.2 Fix 1 (smallest; ships regardless) +
   1.3 (tagalong) + Stream C 2.1 (preflight on the v0.3
   reference run)
2. Stream A 1.1 multi-symbol API + ADR
3. Stream A 1.2 Fix 2 OR Fix 3 (decision point: pre-Stream-D
   evidence-gated)
4. Stream B docstring extraction (TS → Python → Go)
5. Stream C 2.2 + 2.3 (synthesis-time additions; cost
   prior recalibration)
6. Stream D — v0.3 reference run + Phase 8 synthesis
7. v0.3 ship

Step ordering details land in `STEP-PLAN-V0.3.md` (next
session). Expect 14–18 numbered steps based on this
decomposition.

## Cost envelope

v0.3 splits API spend across two budgets:

**Main-repo extraction budget (~$8–14):**
- Re-extraction runs during Stream A development (4–6
  full self-extractions): ~$4–6
- Stream B docstring-extraction prompt calibration on
  10–15 examples: ~$1–2
- Stream B full extraction runs across hono / httpx /
  cobra (one-time per target): ~$3–6 (depends on
  docstring density per repo; cobra likely cheapest, hono
  most)
- Contingency: ~$2

**Benchmarks-repo reference run (~$15–25):**
- Three v0.3 reference matrices (hono + httpx + cobra) at
  ~$7–10 each per revised Go-aware cost priors
  (Theme 2.3): ~$21–30
- Trim if cobra and httpx come in materially under
  projection per Phase 7 §7

**Combined v0.3 total:** ~$23–39. Comparable to v0.2's
$25–40 envelope. Step 13 (post-v0.3) projects $115–150 per
the Theme 2.3 revised priors.

If the cost-tracking budget warning fires during v0.3 work,
treat that as a signal to investigate rather than raise the
cap.

## Success criteria

v0.3 ships when:

1. **Stream A complete:** Theme 1.2 Fix 1 shipped (ADR
   authoring validation); Theme 1.2 Fix 2 OR Fix 3 shipped
   per Stream D pre-run evidence (or the rationale for
   neither documented in the Phase 8 synthesis); Theme 1.1
   multi-symbol API shipped with new ADR; Theme 1.3 atlas
   schema v1.3 with `contextatlas_commit_sha` shipped.

2. **Stream B docstring source landed across three
   languages:** TS / Python / Go docstring extraction
   producing claims with `source: "docstring:<path>"` in
   atlas output. v0.3 atlases for hono / httpx / cobra

## Rescope conditions

v0.3 rescopes (not kills — rescopes) if:

- **Theme 1.2 Fix 2 + Fix 3 both fail to improve bundle
  precision on Phase 8 measurement.** v0.3 ships Fix 1 only,
  with Phase 8 documenting what Fix 2/3 attempted and why
  evidence didn't support shipping. Either:
  (a) The muddy-bundle problem is downstream of attribution
      mechanics — different fix needed (v0.4 candidate); OR
  (b) Phase 6 §5.1's diagnosis was incomplete. Reopen the
      investigation. Either way, v0.3 doesn't ship a
      precision claim that Phase 8 evidence doesn't support.

- **Stream B docstring extraction surfaces a fundamental
  prompt incompatibility.** If `EXTRACTION_PROMPT` cannot
  reliably extract claims from docstrings without
  per-language prompt forking (one prompt per language
  rather than one shared prompt), the dual-prompt pattern
  becomes a per-language pattern. v0.3 absorbs the
  proliferation OR descopes Stream B to one language pending
  v0.4 redesign. Default: absorb if total prompt code stays
  under ~3 prompts; descope if it would balloon.

- **Stream D reference run contradicts Phase 5/6/7 patterns
  on the sharpened atlas.** If the v0.3 atlas measurably
  *worsens* on win-bucket prompts (CA loses cells that
  previously won), v0.3 stops and investigates. The
  hypothesis would be that Theme 1.2's narrower attribution
  removed signal that was load-bearing. Recovery path:
  per-cell trace inspection, then either narrower-attribution
  rollback or additive correction.

- **Atlas-file-visibility filter (Theme 2.1) surfaces more
  contamination patterns than Phase 7 §5.2 anticipated.** If
  cross-running v0.2 reference data through the filter shows
  >10% of beta cells affected on at least one target, the
  trace-time filter approach is insufficient. Pivot to
  clean-workspace mode (longer-term direction in the
  methodology note); v0.3 ship absorbs the larger work or
  defers it to v0.4 with the documented methodology limit.

- **External-facing target raises evidence bar.** If during
  v0.3 execution the resume/community-sharing target reveals
  a public claim that v0.3 evidence can't support
  rigorously, that claim doesn't ship — even if it would
  have been acceptable for an internal milestone. v0.3 over-
  shipping accuracy is preferable to under-shipping
  credibility.

- **Prompt-set integrity violation discovered.** If during
  v0.3 execution it surfaces that one or more pre-registered
  v0.2 prompts produces ambiguous or invalidated results
  (e.g., another c6-class measurement artifact), and fixing
  the prompt would require breaking pre-registration
  discipline, treat as rescope event. Options: descope
  Stream D to the unaffected language(s); freeze the
  affected prompt and exclude its cells from v0.3-vs-v0.2
  delta tables; or extend v0.3 timeline to author
  replacement prompts under fresh pre-registration. Decision
  belongs to scope-doc author, not Stream D executor — the
  apples-to-apples framing depends on the locks holding, so
  any modification needs scope-level authorization.

Per ROADMAP.md's `Rescope conditions` section — same
philosophy, applied to v0.3 scope.

## Open questions (to resolve during v0.3)

1. **Theme 1.2 Fix 2 vs Fix 3 selection.** Which actually
   fixes the muddy-bundle problem on Phase 6 p4? Stream A
   ships Fix 1 unconditionally; Fix 2 (narrower attribution)
   and Fix 3 (BM25 ranking on get_symbol_context) need
   pre-Stream-D evidence to choose between. Resolution
   trigger: Stream A 1.2 implementation ready for spot-check
   measurement on a single cell before full Phase 8.

2. **Multi-symbol API surface — separate tool or option on
   existing tool?** Two shapes:
   (a) New tool `get_symbols_context([sym1, sym2, ...])`
       alongside `get_symbol_context(sym)`.
   (b) Extend existing `get_symbol_context` to accept
       single-or-array input.
   Default lean: (b) is more idiomatic for MCP, but (a) has
   cleaner discriminator semantics for tool-routing. Resolve
   during ADR-N authoring.

3. **Docstring source-key shape.** `"source": "docstring:<path>"`
   uses path as discriminator. Alternative:
   `"source": "docstring", "source_path": "<path>"` —
   stricter separation between source type and source
   location. Decide during Stream B implementation; affects
   Phase 8 synthesis-doc query examples.

4. **v0.3 → v0.4 trigger.** v0.3 scope explicitly defers
   quality-axis measurement to v0.4. What's the signal that
   v0.4 should start? Candidates:
   - (a) **Evidence-based:** v0.3 efficiency + precision
     measurements suggest a quality differential exists that
     v0.4's blind grading would resolve.
   - (b) **Calendar-based:** A fixed window post-v0.3 ship.
   - **Default preference: (a)**, consistent with
     evidence-gated guiding principle. Calendar fallback if
     evidence doesn't arrive organically within ~2 months
     post-v0.3 ship.

5. **Phase 8 synthesis depth.** The v0.2 retrospective
   established that synthesis-note LOC is finding-scaled,
   not estimate-scaled (Phase 5/6/7 ran 848 / 593 / 540
   LOC). Phase 8 likely runs in the same range; resist
   pre-anchoring at "shorter than Phase 5." Length is what
   Stream D's findings warrant.

6. **External-facing target implementation surface.** What
   specifically does "resume-building + community sharing"
   require beyond v0.3's technical deliverables? Polished
   README? Blog post? Public benchmark site? Resolve before
   v0.3 ship gate so v0.3 doesn't ship missing the
   community-readable-output bar.
